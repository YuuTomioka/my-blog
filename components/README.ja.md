# Components ガイド (`components/`)

このドキュメントは、`my-blog` における `components/` の使い方を定義します。

現在のプロジェクト前提:
- ランタイム: Next.js App Router（`app/`）
- データ層: `lib/` と `content/index/` の事前生成 JSON
- スタイル: `app/globals.css`（Tailwind は未導入）

## 目的

- `app/` 配下のページ実装を小さく保つ
- UI描画とデータ取得・インデックス処理を分離する
- 将来の UI 改修（v2.x）を安全に段階適用できるようにする

## ディレクトリ方針

想定構成:

```text
components/
  plus/   # 原本保管庫（参照専用）
  blog/   # 記事一覧・検索結果などブログ固有の表示コンポーネント
  ui/     # サイト共通のレイアウト/基盤UI
```

補足:
- `components/plus/` は参照用の原本を保管する場所です。
- 実運用で `app/*` から直接参照するのは `blog/` と `ui/` のみとします。
- `plus/` をベースに、プロジェクト向けに改修したものを `blog/` / `ui/` に配置してください。

## 責務

`components/blog/*`
- ブログ領域の表示のみ（PostCard、メタ表示、一覧、検索結果ブロックなど）
- データは props で受け取る
- ファイル/API から直接取得しない

`components/plus/*`
- 参照用の原本コンポーネント（スナップショット）
- 可能な限り原本に近い状態で保持する
- プロジェクトのデータ取得ロジックと結合しない

`components/ui/*`
- ページ横断の共通UI（Container、Section枠、Badge、Empty State など）
- 必要がない限りブログ固有仕様を持ち込まない

`app/*`
- ルート単位のオーケストレーション
- `lib/*` 経由でデータを取得
- 整形済み props を `components/*` に渡す

`lib/*`
- データ取得・整形・検索スコアリング・正規化を担当
- JSX 描画責務は持たない

## 必須ルール

- `components/*` から `content/` を直接読まない
- `components/*` で `fs` / `path` / build script を使わない
- React コンポーネントに export/index ロジックを混ぜない
- コンポーネントは props 駆動で決定的に動作させる（隠れたグローバル状態を持たない）
- アクセシビリティを維持する（`aria-*`、見出し階層、キーボード操作可能性）
- `plus/` 配下を本番実装として直接利用しない

## スタイリング方針（現行）

- まず `app/globals.css` の既存トークン/クラスを再利用する
- 再利用が難しい場合のみグローバルクラスを追加する
- バージョン化された Issue なしに新しい UI フレームワークを導入しない

## 実装フロー

1. `components/plus/*` からベースとなる参照コンポーネントを選ぶ
2. `components/blog/*` または `components/ui/*` に改修版を実装する
3. `app/*` でコンポーネント境界（必要props）を定義して利用する
4. データ整形は `lib/*` に残し、最小限の props を渡す
5. `npm run dev` で表示確認する
6. `npm run index` と `npm run build` でパイプライン整合を確認する

## レビューチェックリスト

- このコンポーネントは表示責務に限定されているか
- props の形が明確か
- データはコンポーネント到達前に整形済みか
- 既存スタイルとセマンティックHTMLを活用しているか
- export/index スクリプトへの結合を増やしていないか
