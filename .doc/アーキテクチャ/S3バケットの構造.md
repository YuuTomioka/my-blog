# 📁 S3フォルダ構造設計ドキュメント（ブログシステム用）

## 🎯 目的

本ドキュメントは、S3バケット上に保存されるブログ記事および関連リソース（画像・固定ページ・添付等）の構造設計方針を示す。主に以下を目的とする：

- Markdownファイルと画像・添付ファイルの**論理的なまとまり**を保つ
- **Next.jsによる参照・配信・キャッシュ制御**を最適化する
- GitHub Actionsによる**自動同期と削除**を安全かつ効率的に行う
- 将来的な**検索・集計・リファクタリングの容易さ**を担保する

---

## 🧱 ベース構成

S3バケットの構造は、以下のように論理分類される：

```
S3 Root
├── post/ # 記事コンテンツ（ブログ投稿）
│ └── YYYY/MM/DD/slug/
│ ├── index.md ← 記事本文（Frontmatter付き）
│ ├── cover.jpg ← サムネイル・OGP画像
│ ├── image-1.png ← 本文内画像
│ └── embeded/... ← 埋め込み素材（図・動画等）
├── page/ # 固定ページ（about, privacyなど）
│ ├── about.md
│ └── contact.md
├── asset/ # 共通素材（ロゴ・背景等）
│ └── common/
│ ├── logo.svg
│ └── favicon.png
```

---

## 📄 Markdownファイル規約

### 命名
- 各記事は `index.md` を使用し、フォルダ単位で記事を識別
- 固定ページは `pages/slug.md` で定義（例：`pages/about.md`）

### Frontmatter（YAML形式）

```yaml
---
title: "記事タイトル"
slug: "hello-world"
date: "2025-09-15"
tags: ["nextjs", "s3"]
status: "published"      # or "draft"
cover: "cover.jpg"       # カバー画像ファイル名
excerpt: "短い紹介文"
---
```

---

## 🖼 画像・添付ファイルルール

- **記事フォルダ内に配置すること**
    - 相対パスでMarkdownから参照（例：`./chart.png`）
- **画像名にスラッグや内容を含める**と整理しやすい（例：`flow-overview.svg`）
- Markdownからの参照例：

```markdown
![フローチャート](./chart-1.png)
![構成図](./embeded/diagram.svg)

```

- **カバー画像**は `cover.jpg` を基本とし、Frontmatterで明示

## 🔁 GitHub Actions対応

### 同期対象

- `content/posts/**/*` → `s3://<bucket>/posts/...`
- `content/pages/**/*` → `s3://<bucket>/pages/...`
- `content/assets/**/*` → `s3://<bucket>/assets/...`

### 同期オプション（例）

```bash
aws s3 sync content/posts s3://$S3_BUCKET/posts \
  --delete \
  --exclude "*/drafts/*" \
  --exclude "*/.*"

```

- **`-delete` により削除も反映**
- `drafts/` や `.obsidian/` などは除外設定で制御

---

## 🌐 CDN（CloudFront）との連携前提

- バケットは **完全非公開**
- CloudFront + OAC（Origin Access Control）経由でのみ公開
- Next.js や Markdownリンクは、以下のように生成：

```
https://cdn.example.com/posts/2025/09/hello-world/chart-1.png

```

または Next.js 側で：

```tsx
const url = `${process.env.NEXT_PUBLIC_ASSET_BASE_URL}/posts/${year}/${month}/${slug}/${filename}`

```

## 📦 その他の設計指針

| 項目 | 内容 |
| --- | --- |
| 削除容易性 | 記事ごとにディレクトリ構造を持つことで `aws s3 rm --recursive` が自然に使える |
| キャッシュ戦略 | CloudFrontのパス単位で分離キャッシュが可能（記事単位の制御） |
| URL整合性 | `.md` からのリンクはローカル環境とS3/CDNで整合する（Obsidian/VScodeでの表示と一致） |
| コンポーネント再利用 | 自己紹介や一部文言を `.md` として参照する場合にも再利用可能 |
| 将来の全文検索/タグ分類 | `tags`, `date`, `slug` をFrontmatterで明示することでメタデータ抽出が容易に |

---

## 🚫 禁止事項・非推奨パターン

| パターン | 理由 |
| --- | --- |
| 画像を記事外の共通 `images/` に保存 | 関連が断たれ、削除・整理・相対参照が難しくなる |
| 複数記事でファイルを共有 | 記事の独立性が崩れ、将来的な分割/移動に支障が出る |
| `.md` だけで記事識別（ファイル名＝スラッグ） | 添付画像や構成拡張に対応しにくい（自己完結性がない） |

---

## 🧪 テスト・バリデーション指針

- Markdownレンダリング時に画像リンク切れがないかを CI等でチェック可能
- 記事スラッグ重複は構造上防止される（パスがユニーク）
- 記事構造変更（移動・削除）は `git mv/rm` により自然に追跡される

---

## 🏁 おわりに

この構造設計は、**S3とGitを組み合わせた自己管理型コンテンツ配信**を長期にわたり安定運用するための中核である。構造が明確であることで、AIによるパース、検索システム構築、タグ/カテゴリ導出、他形式への変換など、将来的な拡張も容易になる。
